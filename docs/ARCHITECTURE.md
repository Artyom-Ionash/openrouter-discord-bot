# Архитектура проекта (Python Bot Edition)

Проект построен по принципам модульной архитектуры, адаптированной для экосистемы Python. Мы сохраняем строгое разделение ответственности, обеспечивая чистоту бизнес-логики и типизацию через `mypy`.

## ⚛️ Ядро (The Core Kernel)

Ядро структурировано по технологическим доменам. Это предотвращает «загрязнение» логики сетевыми API или зависимостями фреймворков.

### Структура `core/`:

1.  **`core/primitives/`** (Universal)
    - **Что это:** Чистые функции, обработка строк, математика, логика маппинга данных.
    - **Зависимости:** 0. Никакого Discord.py или OpenAI API.
    - **Цель:** Легкая тестируемость.

2.  **`core/integrations/`** (Adapters)
    - **Что это:** Обёртки над внешними сервисами (OpenAI, OpenRouter, Discord Gateway).
    - **Цель:** Изоляция внешней нестабильности. Если API изменится, мы меняем код только здесь.

3.  **`core/types/`** (Domain Contracts)
    - **Что это:** Pydantic-модели и TypedDicts, описывающие структуру данных бота.
    - **Цель:** Обеспечить строгую типизацию данных, передаваемых между модулями.

## 🏗 Принципы слоев (Boundaries)

Мы используем `mypy` в режиме `--strict` для защиты границ.

### 1. Feature Layer (`features/`)

- **Назначение:** Полноценные возможности бота (напр. `/chat`, `/commands`).
- **Структура:** Модули, группирующие логику конкретной функции.
- **Правила:**
  - Могут использовать `core/` и `lib/`.
  - Запрещено напрямую вызывать внешние API в обход `core/integrations/`.

### 2. Domain Logic (`lib/`)

- **Назначение:** Основная логика обработки данных.
- **Пример:** `lib/context_manager.py` (сбор истории), `lib/token_calculator.py`.
- **Правила:** Могут использовать `core/primitives/`. **Запрещено** использование JSX (по понятным причинам) и сторонних UI-библиотек.

## 🧠 Принципы проектирования (Design Principles)

### 1. Mining & Refining (Добыча и Обогащение)

Принцип разработки остается неизменным:

1.  **Mining (Exploration):** Быстрая реализация функции в `bot.py` или отдельном скрипте.
2.  **Refining (Exploitation):** Перенос логики в изолированные функции/классы в `lib/` или `core/`, покрытие их тестами (`pytest`).

### 2. Dependency Flow (Гравитация зависимостей)

Поток данных:
`bot.py` (Entry Point) → `features/` → `lib/` → `core/`

- `bot.py` — это только клей (glue code). В нем не должно быть сложной бизнес-логики.
- `lib/` — чистая логика (парсинг истории, расчет токенов).
- `core/` — инфраструктура.

### 3. Type Integrity (Целостность типов)

В Python мы используем `Pydantic` для рантайм-валидации и `mypy` для статической.

- **Запрет на Any:** Использование `Any` допустимо только на границе с внешним API (например, при парсинге JSON), сразу после этого данные должны конвертироваться в `BaseModel`.
- **Type Guards:** Используем функции `isinstance()` и Pydantic `model_validate()`.

## 🛡 Качество кода

- **Strict Mypy:** Все функции должны иметь аннотации типов (`def func(arg: int) -> str:`).
- **Ruff:** Стандарт для линтинга и сортировки импортов. Автоматически исправляет структуру импортов (аналог `simple-import-sort`).
- **Dependencies:** Все зависимости управляются через `pyproject.toml` (через `uv` или `pip`).

---

## 📚 Принципы документации

- **Лаконичность:** Один файл = одна задача.
- **Code as Documentation:** Хорошо типизированный код не требует комментариев.
- **AI-Native:** Всегда генерируйте документацию через скрипты, чтобы поддерживать актуальность контекста для LLM.
